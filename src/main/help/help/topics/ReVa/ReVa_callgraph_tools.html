ReVa Call Graph Analysis Tools

The ReVa call graph tools provide MCP-accessible interfaces for analyzing function calling relationships. These tools support bidirectional graph exploration, hierarchical tree traversal, and identification of common dispatch points. All three tools include cycle detection, depth limiting, and node count caps to prevent runaway analysis on complex binaries.

get-call-graph

Get the call graph around a function, showing both callers (functions that call this one) and callees (functions this one calls) up to a specified depth. This tool provides a quick overview of a function's role in the broader program flow by exploring both directions simultaneously.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
functionAddress string Yes — Address or name of the center function. Accepts hex addresses (e.g., "0x401000") or function names (e.g., "main"). If the address is inside a function body, the containing function is used.
depth integer No 1 Number of levels of callers and callees to include. Automatically clamped to the range 1–10.

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • centerFunction — Object with name and address of the center function.
  • depth — The depth that was used.
  • callerCount — Number of caller nodes returned.
  • calleeCount — Number of callee nodes returned.
  • callers — Nested array of caller node objects.
  • callees — Nested array of callee node objects.

Each node object contains:

  • name — Function name.
  • address — Function address in hex.
  • cyclic — (Present only if true) Indicates a cycle was detected. The node does not recurse further.
  • callers or callees — Nested child nodes at the next depth level.

Graph Behavior

  • Bidirectional: Both caller and callee directions are explored in a single request.
  • Permanent visited tracking: Each function appears at most once per direction. Once a function is visited in the caller tree, it will not appear again in that tree (but may still appear in the callee tree).
  • Node cap: Maximum 250 nodes per direction. If the cap is reached, exploration stops at the current depth level.
  • Cycle detection: Recursive or mutually-recursive functions are marked with "cyclic": true and are not expanded further.

Start with depth=1 (the default) for a quick overview, then increase depth selectively. Higher depths can produce very large result sets on well-connected functions.

get-call-tree

Get a hierarchical call tree starting from a function, traversing in a single direction (callers or callees). Unlike get-call-graph, the tree view allows the same function to appear in different branches, giving a complete picture of all call paths.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
functionAddress string Yes — Address or name of the root function.
direction string No "callees" Direction to traverse: "callers" (upward — who calls this function) or "callees" (downward — what this function calls). Case-insensitive.
maxDepth integer No 3 Maximum depth to traverse. Automatically clamped to the range 1–10.

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • direction — The direction traversed ("callers" or "callees").
  • maxDepth — The maximum depth that was used.
  • totalNodes — Total number of nodes in the tree.
  • tree — The root tree node.

Each tree node contains:

  • name — Function name.
  • address — Function address in hex.
  • depth — Current depth in the tree (0 for the root).
  • cyclic — (Present only if true) Indicates a cycle was detected within the current call path. The node is not expanded further.
  • truncated — (Present only if true) Indicates the maximum depth was reached. No children are listed.
  • callers or callees — Array of child tree nodes (matching the traversal direction).

Tree vs. Graph Behavior

Aspect Graph (get-call-graph) Tree (get-call-tree)
Directions Both directions simultaneously One direction at a time
Visited tracking Permanent (per direction) — each function appears once Per-path with backtracking — same function can appear in different branches
Duplicate nodes No duplicates within a direction Duplicates allowed across branches
Node limit 250 per direction 500 total
Best for Quick overview of a function's context Deep call path analysis and hierarchy

Because the tree allows duplicates across branches, it has a higher node limit (500) than the graph view (250 per direction). Cycle detection in tree mode operates per-path: a function is only marked cyclic if it appears in its own ancestor chain, not if it appears elsewhere in the tree.

find-common-callers

Find functions that call all of a set of specified target functions. This tool identifies shared dispatch points, main loops, or common entry points that orchestrate multiple related functions. It is particularly useful for finding the function that ties together a set of related operations.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
functionAddresses array of strings Yes — List of function addresses or names to find common callers for. At least one address is required. Example: ["main", "0x401200", "parse_input"]

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • targetFunctions — Array of resolved target function objects, each with name and address.
  • commonCallerCount — Number of common callers found.
  • commonCallers — Array of common caller objects, each with name and address. Sorted by address in ascending order.

Algorithm

The tool uses set intersection to find common callers:

  1. Collect all direct callers of the first target function.
  2. For each subsequent target function, intersect its callers with the running set.
  3. If the intersection becomes empty at any point, the operation terminates early and returns an empty result.

Only direct callers are considered. The tool does not traverse multiple levels of the call graph. To find indirect common callers, combine this tool with get-call-tree in the "callers" direction.

Error Conditions

  • An empty functionAddresses array produces the error: "At least one function address is required".
  • If an address cannot be resolved, the error includes the unresolvable address string.
  • If no function exists at a resolved address, the error includes the address.

This tool is especially useful when you have identified several related utility functions (e.g., encryption, networking, file I/O) and want to find the higher-level function that orchestrates them all.

Global Configuration

The following limits and timeouts apply across all call graph tools:

Setting Value Description
Maximum depth 10 Hard cap for depth and maxDepth parameters. Values above 10 are clamped.
Graph node cap (per direction) 250 Maximum nodes in each direction of get-call-graph.
Tree node cap (total) 500 Maximum total nodes in get-call-tree.
Operation timeout 60 seconds All call graph operations time out after 60 seconds. The error message is: "Operation cancelled or timed out".

The timeout monitor checks for cancellation before each recursive expansion and before processing each function in a loop. This ensures prompt termination when the time limit is reached.
