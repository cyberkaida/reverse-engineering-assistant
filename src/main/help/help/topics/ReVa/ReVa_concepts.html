ReVa Architecture & Concepts

This page describes the internal architecture, communication protocols, and key design decisions behind the ReVa (Reverse Engineering Assistant) extension. Understanding these concepts will help you work with ReVa more effectively and troubleshoot any issues that arise.

The Model Context Protocol (MCP)

ReVa communicates with AI assistants using the Model Context Protocol (MCP), an open standard for connecting AI models to external tools and data sources. MCP defines how tools are discovered, invoked, and how results are returned, providing a structured interface between the AI assistant and Ghidra's analysis capabilities.

Streamable HTTP Transport

ReVa uses the Streamable HTTP transport, which provides a modern, session-aware connection over standard HTTP. This is not the older Server-Sent Events (SSE) transport — ReVa uses a forked version of the MCP SDK's streamable transport provider with fixes for session stability.

The transport works as follows:

  1. A client sends an HTTP POST request to the MCP endpoint.
  2. The server processes the request and returns a response, optionally streaming results via chunked transfer encoding.
  3. Sessions are maintained across requests, allowing the server to track client state.

The default MCP endpoint is http://localhost:8080/mcp/message. The port and host are configurable through the ReVa plugin options (default host: 127.0.0.1, default port: 8080).

Multiple Simultaneous Clients

The MCP server supports multiple clients connecting simultaneously. Each client establishes its own session, and all clients have access to the same set of tools and programs. This means you can have multiple AI assistants or scripts connected to the same Ghidra instance at once.

All connected clients see the same programs and share the same Ghidra project state. Changes made by one client (such as renaming a function) are immediately visible to all other clients.

API Key Authentication

ReVa supports optional API key authentication to restrict access to the MCP server. When enabled, clients must include a valid API key in their requests. API keys use the format ReVa-{UUID} and can be configured in the ReVa plugin options.

By default, API key authentication is disabled, and the server only listens on 127.0.0.1 (localhost), so it is not accessible from other machines on the network.

Plugin Architecture

ReVa consists of two Ghidra plugins that work together to provide MCP functionality throughout the Ghidra session. Understanding how these plugins interact is important for troubleshooting and for understanding when programs become available to AI assistants.

RevaApplicationPlugin (Application-Level)

The RevaApplicationPlugin is an application-level plugin that runs in Ghidra's front-end tool. It is loaded once when Ghidra starts and persists for the entire Ghidra session. This plugin:

  • Owns and manages the MCP server (start, stop, restart)
  • Manages the RevaProgramManager, which tracks all open programs across all Ghidra tools
  • Provides the RevaMcpService that tool-level plugins connect to
  • Listens for project open/close events

Because this plugin lives at the application level, the MCP server starts when Ghidra starts and remains running even when no CodeBrowser windows are open. The server will be available as long as Ghidra itself is running.

RevaPlugin (Tool-Level)

The RevaPlugin is a tool-level plugin that runs in each CodeBrowser window. Each time you open a new CodeBrowser, a new instance of RevaPlugin is created. This plugin:

  • Connects to the application-level RevaMcpService
  • Registers its CodeBrowser tool with the MCP server
  • Notifies the server when programs are opened or closed in its CodeBrowser
  • Provides UI components (if any) in the CodeBrowser window

Startup Sequence

The following sequence occurs when Ghidra starts and a program is opened:

  1. Ghidra starts → RevaApplicationPlugin initializes
  2. MCP server starts → Listening on configured host and port
  3. CodeBrowser opens → RevaPlugin initializes, connects to RevaMcpService
  4. Program opens in CodeBrowser → RevaPlugin notifies server, program becomes available to AI clients

When a program is closed in the last CodeBrowser that had it open, the server notifies all tool providers that the program is no longer available.

Tool Providers

The MCP server registers 17 specialized tool providers, each exposing a set of related tools to AI clients. The tool providers are organized into categories:

Category Tool Providers Description
Core Analysis Decompiler, Functions, Strings, Symbols, Cross-References, Memory Primary analysis tools for understanding program behavior
Data & Types Data, Data Types, Structures Working with program data and type definitions
Advanced Analysis Call Graph, Data Flow, Constants, VTable, Imports/Exports Deeper analysis capabilities for complex investigations
Annotations Comments, Bookmarks Adding notes and markers to the program database
Project Management Project Managing programs, files, and version control

Program Identification

Every tool in ReVa that operates on a program requires a programPath parameter to identify which program to work with. Understanding how program identification works is essential for using ReVa effectively.

What is a programPath?

A programPath is the Ghidra project pathname of a program. It looks like a file system path within the Ghidra project:

  • /Hatchery.exe — A program at the root of the project
  • /malware/sample_001.bin — A program in a subfolder
  • /firmware/bootloader.elf — Another nested program

The programPath is not the file system path where the original binary lives. It is the path within the Ghidra project, as shown in the Ghidra Project Manager window.

How to Obtain a programPath

There are three ways to discover the programPath for a program:

  1. get-current-program — Returns the program currently active in the focused CodeBrowser window (GUI mode only). The response includes the programPath field.
  2. list-project-files — Lists all files and folders in the Ghidra project. Each file entry includes a programPath field that can be used directly as input to other tools.
  3. list-open-programs — Lists all programs currently open across all CodeBrowser windows (GUI mode only).

The programPath returned by any of these tools can be passed directly to any other ReVa tool that requires it.

Program Resolution

When you provide a programPath to a tool, ReVa resolves it using a multi-tier lookup strategy:

  1. Registered programs — Programs explicitly registered (test environments)
  2. Program cache — Programs recently accessed, keyed by canonical domain path
  3. Tool manager lookup — Active programs in running CodeBrowser tools
  4. Domain file opening — Opens the program directly from the project files on demand

If the program cannot be found, the error message will include suggestions of available programs to help you correct the path.

Three Operational Modes

ReVa can operate in three distinct modes, each suited to different workflows. All three modes share the same core tool providers and MCP server implementation, ensuring consistent behavior regardless of how you connect.

Assistant Mode (GUI)

This is the primary mode when running ReVa as a Ghidra plugin with the full graphical interface. The MCP server starts automatically when Ghidra launches.

Endpoint: http://localhost:8080/mcp/message
Transport: Streamable HTTP
Programs: Any program open in any CodeBrowser window
Configuration: Ghidra Tool Options (persisted in Ghidra settings)

In this mode, you interact with Ghidra normally while an AI assistant connects to the MCP endpoint. You can open and close programs in CodeBrowser windows, and they will automatically become available to (or removed from) the AI assistant.

Headless Mode

Headless mode runs Ghidra without a graphical interface, using PyGhidra to initialize the Ghidra framework. This is useful for scripting, automation, and server deployments.

Endpoint: http://localhost:8080/mcp/message (configurable)
Transport: Streamable HTTP
Programs: Loaded programmatically or via import tools
Configuration: File-based or in-memory configuration backend

To start headless mode:

python scripts/reva_headless_server.py --wait

Some GUI-only tools (get-current-program, list-open-programs) are not available in headless mode, as there is no active CodeBrowser window.

CLI/Stdio Mode

CLI mode provides a stdio-based MCP transport for direct integration with AI tools like Claude CLI. The mcp-reva command handles the full lifecycle: initializing PyGhidra, starting a local HTTP server on a random port, and bridging stdio to HTTP.

Command: mcp-reva
Transport: Stdio (bridged to internal HTTP)
Programs: Imported via tools, stored in .reva/projects/
Configuration: Minimal (random port, auto-configured)

To add ReVa to Claude CLI:

claude mcp add ReVa -- mcp-reva

In this mode, temporary Ghidra projects are automatically created and cleaned up. Programs are imported using the import-file tool and stored in a .reva/projects/ directory.

The Read-Before-Modify Pattern

ReVa enforces a read-before-modify pattern for certain decompiler-based tools. Before an AI assistant can rename variables or change variable data types in a function, it must first read the function's decompilation using the get-decompilation tool.

Design Rationale

This pattern exists for an important reason: it prevents blind modifications. Without this requirement, an AI assistant could rename variables or change types without having seen the current state of the function. By requiring a read first, ReVa ensures that:

  • The AI has seen the current variable names, types, and decompiled code
  • Modifications are informed by the actual state of the function
  • There is a clear audit trail of what was examined before changes were made

30-Minute Expiry

The read tracking has a 30-minute expiry window. After reading a function's decompilation, you have 30 minutes to make modifications. After that, you must read the function again before making further changes. This ensures that the AI is always working with reasonably current information.

The following tools require a prior get-decompilation call:

  • rename-variables — Rename local variables and parameters
  • change-variable-datatypes — Change variable data types

The read tracking is automatically cleared when a program is closed. If a program is closed and reopened, you must read its functions again before modifying them.

Pagination

Programs can contain thousands of functions, symbols, strings, and other items. To handle large datasets efficiently, all list-style tools in ReVa support pagination using two standard parameters:

Parameter Type Description
startIndex integer The zero-based index to start from (default: 0)
maxCount integer The maximum number of items to return per page

Each paginated response includes metadata indicating the total number of items available, so you can determine how many pages remain. Default page sizes vary by tool (typically 100 items), but can be adjusted using maxCount.

For large programs with thousands of functions, always use pagination. Requesting all items at once can produce very large responses that may be slow to process.

Error Handling

ReVa tools return structured JSON responses for both success and error cases. When an error occurs, the response will contain an error field with a descriptive message.

Common Error Types

Error Cause Resolution
Program not found The programPath does not match any open or available program Use list-project-files or list-open-programs to find the correct path. The error message includes suggestions.
Function not found The specified address does not contain a function Use list-functions to find valid function addresses
Read required Attempting to modify a function without reading it first Call get-decompilation on the function before modifying
Decompilation failed The decompiler could not process the function (timeout or error) The function may be too complex. Check the error message for details.
Missing parameter A required parameter was not provided Check the tool description for required parameters

Automatic Error Wrapping

All tool handlers are wrapped with automatic exception handling. Common exceptions like IllegalArgumentException (missing or invalid parameters) and ProgramValidationException (program not found) are automatically converted to structured error responses. This ensures that AI clients always receive a parseable response, even when errors occur.

Performance Limits

Several tools have built-in limits to prevent excessive computation or overly large responses. Understanding these limits helps you plan your analysis effectively.

Tool Limit Value Notes
Call Graph Max nodes per direction 250 Graph view; tree view allows up to 500 nodes. Max depth: 10.
Data Flow Max slice size 500 SSA operations Limits the program slice to prevent excessive analysis
Constant Search Instruction scan limit 2,000,000 instructions Hard cap: 10,000 results. Default: 500 results.
Decompiler Timeout 10 seconds (configurable) Per-function decompilation timeout
Import Analysis Timeout 600 seconds Auto-analysis after import
Cross-References Default page size 100 (max 1,000) Per-request limit for cross-reference queries
Decompiler Callers Per response / total 50 / 500 Max callers or callees shown per response; max collected in bulk

The decompiler timeout is configurable through the ReVa plugin options. Increase it for programs with very complex functions. All other limits are fixed safety bounds designed to keep responses fast and manageable.
