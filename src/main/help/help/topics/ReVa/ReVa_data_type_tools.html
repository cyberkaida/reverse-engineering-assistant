ReVa - Data Type & Structure Tools

The Data Type and Structure tools provide comprehensive capabilities for browsing, creating, validating, and managing data types and structures within a Ghidra program. These tools expose Ghidra's data type system through the MCP protocol, enabling AI-assisted type annotation and structure definition.

Data Type Browsing Tools

These tools allow you to explore and query the data type archives available to a program, including built-in types, program-defined types, and standalone archives.

get-data-type-archives

Retrieves all data type archives available for a specific program. Archives are returned in priority order: built-in types first, then the target program's types, followed by other open programs and standalone archives.

Parameters

NameTypeRequiredDescription
programPath string Yes Path in the Ghidra Project to the program (e.g., "/Hatchery.exe").

Return Value

A multi-result response. The first object contains metadata:

FieldTypeDescription
countintegerTotal number of archives returned.

Subsequent objects each describe one archive:

FieldTypeDescription
namestringArchive name.
typestringArchive type: "BUILT_IN", "PROGRAM", or the manager type.
idstringUniversal ID of the archive (may be null for built-in).
dataTypeCountintegerTotal number of data types in the archive.
categoryCountintegerTotal number of categories.
programPathstringPresent only for "PROGRAM" type archives.

Archive Priority Order

  1. Built-in data type manager — Always included first. Contains fundamental types (int, char, void, etc.).
  2. Target program's data type manager — Types defined within the program itself.
  3. Other open programs — Data type managers from other programs currently open in the project (duplicates are skipped).
  4. Standalone archives — Archives available through Ghidra's DataTypeArchiveService (only available in GUI mode with plugin support).

In headless mode, standalone data type archives from the DataTypeArchiveService may not be available.

get-data-type-by-string

Resolves a data type from its string representation. This is the primary way to look up types using familiar C-style syntax, including pointers, arrays, and composite types.

Parameters

NameTypeRequiredDefaultDescription
programPath string Yes — Path in the Ghidra Project to search for data types.
dataTypeString string Yes — String representation of the data type.
archiveName string No "" (all archives) Name of a specific archive to search. If empty, all archives are searched.

Supported String Formats

  • Basic types: int, char, float, double, byte, short, long
  • Pointers: char*, int**, void*
  • Arrays: int[10], char[256]
  • Complex types: Custom structures and typedefs by name

Return Value

A single JSON object with full data type metadata (same fields as individual items from get-data-types): name, displayName, size, categoryPath, alignment, etc.

If the type cannot be found or parsed, an error is returned. In headless mode without the plugin, the DataTypeArchiveService may not be available, which limits search scope.

Structure Management Tools

These tools provide full lifecycle management for structures and unions: creation from C definitions, validation, inspection, application to memory, listing, and deletion.

parse-c-structure

Parses a C-style structure or union definition and creates it in the program's data type manager. If a structure with the same name already exists, it is completely replaced — all existing fields are cleared and rebuilt from the new definition.

Parameters

NameTypeRequiredDefaultDescription
programPath string Yes — Path of the program.
cDefinition string Yes — C-style structure definition (e.g., "struct MyStruct { int x; char* name; };").
category string No "/" Category path where the structure will be placed.

Return Value

FieldTypeDescription
messagestringSuccess or modification message.
numComponentsintegerNumber of fields in the structure.
namestringStructure name.
displayNamestringFull display name.
sizeintegerTotal size in bytes.
categoryPathstringCategory location.
isUnionbooleanWhether the type is a union.
isPackedbooleanWhether packing is enabled.
fieldsarrayDetailed field information (for new structures).

Behavior

  • Create-or-replace semantics: If a structure with the same name already exists, its fields are completely cleared and rebuilt from the new definition.
  • Field type resolution: Field types are resolved in the program's DataTypeManager.
  • Bitfield support: Bitfield components are handled with BitFieldDataType.
  • Category creation: The specified category path is created if it does not exist.
  • Description preservation: Descriptions from the parsed definition are copied to the created structure.

Use get-structure-info to inspect the current layout before modifying an existing structure. This tool operates within a Ghidra transaction and rolls back automatically on failure.

Undefined Byte Condensing

Consecutive undefined bytes in a structure are merged into a single entry with the name <undefined>. The entry shows the offset range, total length, and is marked with isCondensed: true. This produces a cleaner, more readable representation of structures with padding or unanalyzed regions.

Parameters

NameTypeRequiredDefaultDescription

Return Value

FieldTypeDescription

The data type must be a structure or union. The target address must be in valid memory. When clearExisting is true (the default), any existing data at the address is cleared before the structure is applied. The operation runs within a Ghidra transaction and rolls back on failure.

delete-structure

Deletes a structure from the program's data type manager. Before deletion, the tool checks for references to the structure across function signatures, variables, and memory locations.

Parameters

NameTypeRequiredDefaultDescription
programPath string Yes — Path of the program.
structureName string Yes — Name of the structure to delete.
force boolean No false Force deletion even if the structure is referenced elsewhere.

Return Value (No References or force=true)

FieldTypeDescription
messagestringSuccess message.
deletedbooleantrue
hadReferencesbooleanWhether the structure had references.
referencesClearedintegerNumber of references that were present.

Return Value (References Found, force=false)

FieldTypeDescription
canDeletebooleanfalse
deletedbooleanfalse
warningstringDescriptive warning message.
referencesobjectDetails of where the structure is referenced.

The references object contains:

FieldTypeDescription
countintegerTotal reference count.
functionsarrayFunction references (format: "FunctionName (return type|parameter: paramName|variable: varName)").
memoryLocationsarrayAddresses where the structure is applied in memory.

Reference Scanning

The tool scans for references in the following locations:

  • Function return types — Functions that return this structure type.
  • Function parameters — Parameters typed with this structure.
  • Function local variables — Local variables using this structure.
  • Memory instances — Defined data in memory that uses this structure.

When references are found and force is false, the structure is not deleted. Review the reference list and either update the references first or use force=true to delete regardless.

parse-c-header

Parses an entire C header file and creates all structures and types defined within it. Uses a two-stage parsing strategy for maximum robustness.

Parameters

NameTypeRequiredDefaultDescription
programPath string Yes — Path of the program.
headerContent string Yes — The full C header file content as a string.
category string No "/" Category path where created types will be placed.

Return Value

FieldTypeDescription
messagestring"Successfully parsed header file"
createdCountintegerNumber of types successfully created.
createdTypesarrayArray of structure info objects for each created type.

Two-Stage Parsing Strategy

  1. Whole file parse — Attempts to parse the entire header content at once using Ghidra's CParser. This is the preferred path as it handles inter-type dependencies correctly.
  2. Fallback line-by-line — If the whole file parse fails or returns null, the parser falls back to accumulating lines until a semicolon is found, then parsing each definition individually. Failed definitions are logged as warnings but do not prevent other definitions from being parsed.

Behavior

  • All parsing and creation occurs within a single Ghidra transaction.
  • The specified category path is created if it does not exist.
  • Each parsed type is resolved into the program's DataTypeManager using the REPLACE_HANDLER conflict handler.
  • Types are moved to the specified category after resolution.
  • The fallback parser is graceful: it creates as many types as possible even when some definitions fail to parse.

This tool is ideal for importing entire header files (e.g., from SDK documentation or reverse-engineered structures). For single structure definitions, parse-c-structure is more appropriate.

Tips & Best Practices

  • Use get-data-type-archives first to discover available archives, then get-data-types to browse their contents.
  • Use get-data-type-by-string when you know the type name and want to quickly resolve it without browsing.
  • Always use validate-c-structure before parse-c-structure when working with complex or untested definitions.
  • Use get-structure-info to inspect existing structure layouts before replacing them with parse-c-structure.
  • Check delete-structure without force=true first to see all references before deciding to proceed with deletion.
  • For large header files, check the createdCount in the parse-c-header response to verify all expected types were created.
  • Paginate through large type lists using startIndex and maxCount parameters.
