ReVa Decompiler & Code Analysis Tools

The ReVa decompiler tools provide MCP-accessible interfaces for decompiling functions, searching decompiled output, renaming variables, changing data types, setting comments, and bulk-decompiling callers or referencers. These tools form the foundation of AI-assisted reverse engineering analysis in ReVa.

All tools that modify decompiler output (rename-variables, change-variable-datatypes, set-decompilation-comment) enforce a read-before-modify requirement. You must call get-decompilation on the target function before modifying it. The read tracking expires after 30 minutes.

get-decompilation

Decompile a function and return its C-like source code with optional metadata. This is the primary analysis tool and the starting point for most reverse engineering workflows.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program containing the function (e.g., "/Hatchery.exe").
functionNameOrAddress string Yes — Function name, address, or symbol. Accepts hex addresses (e.g., "0x00401000"), function names (e.g., "main"), or symbol names (e.g., "start"). Addresses inside a function resolve to the containing function.
offset integer No 1 Line number to start reading from (1-based). Use with limit for pagination of long functions.
limit integer No 50 Number of lines to return. Defaults to 50 to conserve context window space.
includeDisassembly boolean No false Include the assembly listing synchronized with each decompilation line. Produces a synchronizedContent array mapping decompiled lines to their corresponding assembly instructions.
includeComments boolean No false Include comments (pre, eol, post, plate, repeatable) in the output. When includeDisassembly is also true, comments are embedded within the synchronized content.
includeIncomingReferences boolean No true Include incoming cross-references on the function declaration line. Limited to 10 references maximum.
includeReferenceContext boolean No true Include code context snippets from calling functions. Requires includeIncomingReferences to be true.
includeCallers boolean No false Include the list of functions that call this one, with name, address, signature, and call count. Limited to 50 callers.
includeCallees boolean No false Include the list of functions this one calls, with name, address, signature, and call count. Limited to 50 callees.
signatureOnly boolean No false Return only the function signature and metadata without the decompiled body. Use to save context when you only need the prototype.

Return Value

Returns a JSON object containing:

  • programName, functionName, address — Identifying information for the decompiled function.
  • resolvedFrom — How the function was found: "address", "symbol|name-exact", "name-case-insensitive", or "undefined-function".
  • metadata — Function metadata including signature, returnType, callingConvention, isExternal, isThunk, and parameterCount.
  • parameters — Array of parameter objects with name, dataType, and ordinal.
  • startAddress, endAddress, sizeInBytes — Function bounds.
  • totalLines, offset, limit — Pagination state for the decompiled output.
  • decompilation — The decompiled C source code with line numbers.
  • synchronizedContent — (If includeDisassembly) Array of objects mapping each decompilation line to its assembly instructions and comments.
  • incomingReferences — (If includeIncomingReferences) Array of cross-reference objects with fromAddress, fromFunction, callerContext, and callLineNumber.
  • callers, callees — (If requested) Arrays of calling or called function objects with name, address, signature, and callCount.

Undefined Function Support

This tool can decompile addresses that are not yet defined as functions. When an undefined function is decompiled, the response includes isUndefinedFunction: true and a note explaining that the function has not been formally created. Modification tools (rename-variables, change-variable-datatypes) cannot operate on undefined functions; use create-function first.

Read-Before-Modify Tracking

Each successful decompilation is recorded for the read-before-modify system. The tracking key is the combination of programPath and function address, and it expires after 30 minutes. Tracking records are automatically removed when the program is closed.

For large functions, use offset and limit to page through the output rather than retrieving the entire function at once. This conserves context window space when working with AI assistants.

search-decompilation

Search for a regular expression pattern across the decompiled output of all functions in a program. Use this tool to find specific code patterns, API calls, string references, or algorithm implementations across the entire binary.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program to search.
pattern string Yes — Regular expression pattern to search for in decompiled function output.
maxResults integer No 50 Maximum number of matching results to return.
caseSensitive boolean No false Whether the regex search should be case-sensitive.
overrideMaxFunctionsLimit boolean No false Override the configured maximum function limit for large programs. The search may take a long time on binaries with many functions.

Return Value

Returns a JSON object containing:

  • programName — Name of the searched program.
  • pattern — The regex pattern that was used.
  • caseSensitive — Whether the search was case-sensitive.
  • resultsCount — Number of matches found.
  • results — Array of match objects, each containing functionName, functionAddress, lineNumber, lineContent, matchStart, matchEnd, and matchedText.

Limits and Performance

The maximum number of functions searched is controlled by the MAX_DECOMPILER_SEARCH_FUNCTIONS configuration setting. For programs with more functions than this limit, set overrideMaxFunctionsLimit to true. The search sends progress notifications every 10 functions and gracefully continues past functions that fail to decompile or time out.

This operation decompiles each function individually, which can be slow on large binaries. Consider narrowing your search using other tools (such as string search or cross-references) before resorting to a full decompilation search.

rename-variables

Rename one or more high-level variables in a decompiled function. The changes are persisted to the Ghidra program database and a unified diff of the decompilation is returned showing the before and after state.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program containing the function.
functionNameOrAddress string Yes — Function name, address, or symbol identifying the function to modify.
variableMappings object Yes — A JSON object mapping current variable names to new names. Example: {"local_10": "buffer", "param_1": "argc"}

Return Value

Returns a JSON object containing:

  • variablesRenamed — Boolean indicating success.
  • renamedCount — Number of variables renamed.
  • changes — A diff object with hasChanges, summary, additions, deletions, and modifications fields showing the effect on the decompiled output.

Requirements

  • Read-before-modify: You must call get-decompilation on the target function within the last 30 minutes.
  • Defined function required: Cannot rename variables in undefined functions. Use create-function first.
  • All changes are wrapped in a Ghidra transaction and rolled back on error.
  • Variable changes are persisted using HighFunctionDBUtil.updateDBVariable().

change-variable-datatypes

Change the data types of one or more variables in a decompiled function. This modifies the Ghidra program database and returns a unified diff of the decompilation showing before and after state.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program containing the function.
functionNameOrAddress string Yes — Function name, address, or symbol identifying the function to modify.
datatypeMappings object Yes — A JSON object mapping variable names to new data type strings. Example: {"local_10": "char*", "param_1": "int[10]"}. Supports standard C type syntax including pointers and arrays.
archiveName string No "" Optional data type archive to search. If omitted, all available archives are searched.

Return Value

Returns a JSON object containing:

  • dataTypesChanged — Boolean indicating whether any types were changed.
  • changedCount — Number of variables whose types were changed.
  • errors — Array of error strings for any data type changes that failed (e.g., unrecognized type names). Processing continues past errors.
  • changes — A diff object with hasChanges, summary, additions, deletions, and modifications fields.

Requirements

  • Read-before-modify: You must call get-decompilation on the target function within the last 30 minutes.
  • Defined function required: Cannot change types in undefined functions. Use create-function first.
  • Data type strings are parsed by DataTypeParserUtil, which supports C-style notation including pointers (char*), arrays (int[10]), and qualified names.
  • Errors on individual type changes are collected and returned; the tool continues processing the remaining mappings.

set-decompilation-comment

Set a comment at a specific line in a function's decompiled output. The comment is placed at the address corresponding to the specified decompilation line number.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program containing the function.
functionNameOrAddress string Yes — Function name, address, or symbol identifying the function.
lineNumber integer Yes — Line number in the decompiled output (1-based). Obtain line numbers from the get-decompilation output.
commentType string No "eol" Type of comment to set: "pre" (before the line) or "eol" (end-of-line).
comment string Yes — The comment text to set at the specified line.

Return Value

Returns a JSON object containing:

  • success — Boolean indicating success.
  • functionName — Name of the function.
  • lineNumber — The line number where the comment was placed.
  • address — The address corresponding to that decompilation line.
  • commentType — The type of comment set ("pre" or "eol").
  • comment — The comment text that was set.

Requirements

  • Read-before-modify: You must call get-decompilation on the target function first.
  • Defined function required: Cannot set comments on undefined functions.
  • Line numbers map to addresses via Clang token analysis of the decompiler AST.

Use "pre" comments for block explanations before a section of code, and "eol" comments for brief annotations on individual statements.

get-callers-decompiled

Decompile all functions that call a target function in bulk. Returns the full decompilation of each calling function with optional highlighting of the call site. Use this tool to understand how a function is used across the program.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
functionNameOrAddress string Yes — Target function name or address to find callers for.
maxCallers integer No 10 Maximum number of calling functions to decompile per page. Capped at 50.
startIndex integer No 0 Starting index for pagination (0-based). Use nextStartIndex from a previous response to retrieve the next page.
includeCallContext boolean No true Highlight the line containing the call to the target function in each decompilation.

Return Value

Returns a JSON object containing:

  • targetFunction, targetAddress — The function whose callers were decompiled.
  • totalCallers — Total number of unique calling functions found.
  • startIndex, returnedCount, nextStartIndex, hasMore — Pagination state.
  • callers — Array of caller objects, each containing functionName, address, callSites (array of call site addresses), success, decompilation (the full decompiled source), and callLineNumbers (if includeCallContext is true).

Limits

  • Maximum per page: 50 callers per request.
  • Maximum total callers collected: 500 unique callers. If a function has more than 500 callers, only the first 500 are available for pagination.
  • Each decompiled caller is automatically tracked for the read-before-modify system.

get-referencers-decompiled

Decompile all functions that reference a specific address or symbol in bulk. Unlike get-callers-decompiled which only finds call references, this tool also finds data references (reads and writes) to the target.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
addressOrSymbol string Yes — Target address or symbol name (e.g., "0x00401000", "global_var", "my_label").
maxReferencers integer No 10 Maximum number of referencing functions to decompile per page. Capped at 50.
startIndex integer No 0 Starting index for pagination (0-based).
includeDataRefs boolean No true Include data references (reads and writes), not only call references.
includeRefContext boolean No true Include the line numbers in each decompilation where the reference occurs.

Return Value

Returns a JSON object containing:

  • targetAddress, resolvedFrom — The resolved address and how it was resolved.
  • totalReferencers — Total number of unique referencing functions found.
  • startIndex, returnedCount, nextStartIndex, hasMore — Pagination state.
  • referencers — Array of referencer objects, each containing functionName, address, references (array of reference detail objects with fromAddress, refType, isCall, isData, isRead, isWrite), success, decompilation, and referenceLineNumbers (if includeRefContext is true).

Limits

  • Maximum per page: 50 referencing functions per request.
  • Maximum total referencers collected: 500 unique referencing functions.
  • Use includeDataRefs=false to restrict results to call references only, which is equivalent to get-callers-decompiled behavior.
