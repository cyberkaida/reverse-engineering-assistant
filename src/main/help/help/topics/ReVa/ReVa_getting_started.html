Getting Started

This page walks through the first steps of using ReVa for AI-assisted reverse engineering. It assumes ReVa is already installed and activated. If not, see Installation and Setup first.

Opening a Program

Before an MCP client can analyze a binary, the binary must be open in Ghidra with ReVa activated:

  1. Open Ghidra and select (or create) a project.
  2. Import a binary into the project if it is not already present. Use FileImport File from the Project window, or drag the file into the project panel.
  3. Double-click the imported program to open it in the CodeBrowser.
  4. If prompted, allow Ghidra to run auto-analysis. ReVa works best when analysis is complete, as tools such as the decompiler and cross-reference finder depend on the analysis results.
  5. Verify ReVa is active in the CodeBrowser (see Plugin Activation).

The program is now visible to MCP clients. Multiple programs can be open simultaneously across different CodeBrowser windows, and all are accessible through the same MCP server.

In CLI mode (mcp-reva), programs are imported through the import-file tool rather than the Ghidra GUI. The CLI creates a temporary project automatically.

Connecting an MCP Client

With ReVa running and a program open, connect an MCP client to begin analysis. The following example uses Claude Code, but any MCP-compatible client works.

  1. Ensure the ReVa server is running. In GUI mode, the server starts automatically when Ghidra launches with ReVa activated.
  2. Configure the MCP client to connect to ReVa. For Claude Code with a direct HTTP connection, add the following to the MCP configuration: 
    {
  "mcpServers": {
    "ReVa": {
      "url": "http://localhost:8080/mcp/message"
    }
  }
}
  3. Start a conversation with the AI assistant. It now has access to all of ReVa's tools and can interact with the open program.

To verify the connection, ask the assistant to list open programs. It will call the list-open-programs tool and return the available programs with their paths and architectures.

Your First Analysis Session

The following examples demonstrate common analysis tasks. Each example shows a natural-language query and describes the tools the assistant will use to answer it.

Getting an Overview

Start by asking the assistant for a high-level overview of the binary:

"What does this program do? Give me an overview of its functionality."

The assistant will typically:

  1. Call list-open-programs to identify available programs and their architectures.
  2. Call get-strings to survey string references for clues about functionality.
  3. Call list-imports to examine library dependencies and API usage.
  4. Call list-exports to identify the program's public interface.
  5. Call get-functions to survey named functions.
  6. Decompile key functions with get-decompilation to understand their behavior.

Analyzing a Function

Ask about a specific function by name or address:

"Decompile the main function and explain what it does."

"What does the function at 0x00401000 do?"

The assistant will decompile the function, analyze its logic, and explain the behavior. It may follow cross-references to understand called subroutines and data accesses.

Renaming and Annotating

Ask the assistant to improve the database as it understands the program:

"Rename the variables in this function to meaningful names based on their usage."

"Add a comment to 0x00401050 explaining that this is the packet parsing routine."

These operations modify the Ghidra program database. The changes appear immediately in the CodeBrowser and persist when the program is saved. All modifications can be undone with EditUndo.

Tracing Data Flow

Ask more complex analytical questions that require multiple tool calls:

"Where does the user input from recv() end up? Trace the data flow forward."

"What functions call the encryption routine, and what keys do they pass?"

The assistant will combine cross-reference lookups, data flow tracing, call graph traversal, and decompilation to answer these multi-step questions.

Defining Structures

Ask the assistant to create and apply data types:

"This function operates on a network packet. Define a C structure for the packet format based on how the fields are accessed."

The assistant will analyze field accesses in the decompiled code, define a structure using parse-c-structure, and apply it to the relevant variables with apply-structure.

Understanding programPath

Every ReVa tool that operates on a program requires a programPath parameter. This identifier tells the server which program to act on, since multiple programs may be open simultaneously.

Format

The programPath is the Ghidra project pathname of the program. It always begins with a forward slash and corresponds to the program's location in the Ghidra project folder hierarchy.

Scenario programPath
Program at project root /Hatchery.exe
Program in a folder /malware/sample_001.bin
Nested folder /project/firmware/bootloader.elf

Discovering Program Paths

To find the programPath for open programs, use either:

  • The list-open-programs tool — returns all open programs with their paths, architectures, and sizes.
  • The ghidra://programs MCP resource — provides the same information through the resource interface.

The programPath returned by these tools can be passed directly to any other tool that requires it.

If an incorrect or misspelled programPath is provided, the error message will include a list of available programs as suggestions.

Single Program Shortcut

When only one program is open, most MCP clients will automatically use the correct programPath without the analyst needing to specify it. The assistant discovers the path by calling list-open-programs at the start of the session.

Pagination

Tools that return large result sets use pagination to avoid overwhelming the MCP client with data. This applies to tools such as get-functions, get-strings, get-symbols, and others that can return thousands of results.

Pagination Parameters

Paginated tools accept the following parameters:

Parameter Description
offset The starting index (zero-based). Defaults to 0.
limit The maximum number of results to return. Tools define sensible defaults (commonly 100).

Pagination in Responses

Paginated responses include metadata indicating the total number of available results and whether more results exist beyond the current page. The AI assistant handles pagination automatically, requesting additional pages when needed to answer a question.

For large programs with many functions or strings, the assistant may use search and filter parameters to narrow results instead of paginating through the entire list. For example, get-strings accepts a search term, and get-functions can filter by name pattern.

Tips for Effective Prompts

The quality of AI-assisted analysis depends significantly on how questions are framed. The following guidelines help produce better results.

Be Specific About Goals

Specific questions produce better results than vague ones. Compare:

Less Effective More Effective
"Analyze this binary." "Identify the network protocol used by this binary and document the message format."
"What does this function do?" "What cryptographic algorithm does the function at 0x004023A0 implement? Check for known constants."
"Find vulnerabilities." "Check whether the input buffer at 0x00401200 has proper bounds checking before the memcpy call."

Provide Context

Share what is already known about the binary. The assistant cannot read the analyst's mind, but it can use prior knowledge to focus its analysis:

  • "This is a Linux ELF binary from an IoT camera. It appears to implement a custom protocol on port 5000."
  • "I believe the function at 0x00408100 is a hash function based on the constants I see. Can you confirm which algorithm it is?"
  • "This malware sample was caught by our EDR doing DNS tunneling. Help me find the DNS encoding routine."

Work Iteratively

Complex analysis is best done in stages. Start with broad questions to orient, then drill into specific areas of interest:

  1. "List the imported functions. What libraries does this use?"
  2. "I see it imports OpenSSL. Which functions call SSL_read?"
  3. "Decompile the function that calls SSL_read and trace where the received data goes."
  4. "Rename the variables in that function based on your analysis and add a summary comment."

Each step builds on the previous one, and the assistant maintains context across the conversation.

Annotate as You Go

Ask the assistant to annotate the database during analysis rather than only at the end. Renamed functions, typed variables, and comments persist in the Ghidra project and improve subsequent analysis — both for the AI assistant and for human analysts reviewing the work later.

Ask the assistant to use bookmarks to mark locations of interest during analysis. Bookmarks are searchable and appear in Ghidra's Bookmark window, making them easy to review.

Verify and Correct

AI analysis is not infallible. The assistant may misidentify algorithms, misinterpret data structures, or draw incorrect conclusions. Treat the assistant's output as a starting point, not a final answer:

  • Cross-check decompilation output against the disassembly in the CodeBrowser.
  • Verify renamed variables and applied types make sense in context.
  • If the assistant makes a mistake, correct it. The correction improves subsequent analysis within the same session.
  • Use EditUndo to revert any unwanted changes.
