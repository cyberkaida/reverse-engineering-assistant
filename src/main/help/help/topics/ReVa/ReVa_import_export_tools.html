ReVa - Import, Export & Thunk Tools

These tools provide comprehensive access to a binary's imported and exported symbols, including the ability to find cross-references to imports and follow thunk chains to resolve indirect jumps. Understanding a program's imports reveals which external APIs it relies on, while exports show what functionality it provides to other modules.

Import Tools

Tools for listing and searching imported functions from external libraries.

list-imports

Lists all imported functions from external libraries. Results can be grouped by library, filtered by library name, and paginated. This is one of the most useful tools for initial binary triage, as imports reveal the binary's use of system APIs, networking, cryptography, file I/O, and other capabilities.

Parameters

libraryFilter string No — Filter by library name. Uses case-insensitive partial matching (e.g., "kernel32" matches "KERNEL32.dll").
maxResults integer No 500 Maximum number of imports to return. Clamped to range 1–2000.
startIndex integer No 0 Starting index for pagination (0-based).
groupByLibrary boolean No true Whether to group imports by their source library.

Return Value

FieldTypeDescription

When groupByLibrary=true (the default), results are organized under a libraries array:

FieldTypeDescription
libraries[].namestringLibrary name (e.g., "KERNEL32.dll").
libraries[].importCountintegerNumber of imports from this library.
libraries[].importsarrayArray of import objects.

When groupByLibrary=false, results are returned as a flat imports array.

Each import object contains:

FieldTypeDescription
namestringFunction name.
librarystringSource library name.
addressstringAddress in the program (e.g., "0x00401000").
originalNamestringOriginal name if different from current name (optional).
ordinalintegerOrdinal number for ordinal imports (optional).
signaturestringFunction signature if available (optional).

Results are sorted alphabetically by library name, then by function name (case-insensitive). Ordinal imports (e.g., "Ordinal_123") are automatically detected, and the ordinal number is extracted and included in the response.

Export Tools

Tools for listing exported symbols from the binary.

list-exports

Lists all symbols that the binary exports for use by other modules. For DLLs and shared libraries, this shows the public API surface. For executables, this typically shows the entry point.

Each export object contains:

FieldTypeDescription

Exports are identified via Ghidra's external entry point mechanism. For PE files, this correctly identifies all exports registered by the PE loader. Results are sorted alphabetically by name.

Import Reference & Thunk Tools

Tools for tracing how imported functions are called and resolving indirect jump chains.

find-import-references

Finds all locations in the program where a specific imported function is called. Crucially, this tool also follows thunk chains — wrapper functions (IAT stubs) that jump to the actual import — ensuring that indirect calls through thunks are also discovered.

libraryName string No — Specific library name to narrow the search. Case-insensitive. Use when the same function name appears in multiple libraries.
maxResults integer No 100 Maximum number of references to return. Clamped to range 1–500.

Thunk Chain Following

Many compilers generate thunk functions (also called IAT stubs or PLT entries) that simply jump to the actual imported function. A call to CreateFileW might actually call a local thunk that then jumps to the IAT entry. This tool automatically identifies these thunk chains and reports both direct references and references through thunks, with the viaThunk and thunkAddress fields indicating the indirection.

The thunk map is built once per request for efficiency.

If the specified import is not found, an error is returned with the message "Import not found: {importName}". Results are deduplicated by source address.

resolve-thunk

Follows a thunk chain from a given address to find the actual target function. Thunks are wrapper functions that simply jump to another location, often used for import address table (IAT) stubs, PLT entries, or compiler-generated trampolines.

Chain Depth Limit

Thunk chain resolution is limited to a maximum depth of 10 to prevent infinite loops in the case of circular thunk references. If the chain reaches 10 entries without finding a non-thunk target, isResolved will be false.

If no function exists at the specified address, an error is returned: "No function found at address: {address}". The chain always follows one thunk at a time using getThunkedFunction(false) for precise single-step resolution.

Tips & Best Practices

  • Use list-imports with libraryFilter to quickly identify imports from specific libraries of interest (e.g., "ws2_32" for networking, "crypt32" for cryptography, "ntdll" for low-level OS APIs).
  • Use find-import-references to trace how a suspicious API (e.g., VirtualAlloc, CreateRemoteThread) is called throughout the binary.
  • When analyzing a DLL, use list-exports to understand its public API surface before diving into internal functions.
  • Use resolve-thunk when you encounter a function that appears to be a simple jump stub — it will reveal the actual target function.
  • Combine list-imports with groupByLibrary=true for a high-level overview of the binary's dependencies, then use groupByLibrary=false for flat searching across all libraries.
  • For PE binaries with ordinal imports (common in older Windows software), the ordinal number is automatically extracted and included in the import object.
