Installation and Setup

This page covers all methods of installing and configuring ReVa. The most common path is installing the prebuilt extension into Ghidra and connecting an MCP client, but ReVa also supports headless and CLI deployments.

Plugin Installation

ReVa is distributed as a standard Ghidra extension archive (.zip file). To install it:

  1. Download the ReVa extension archive matching the installed Ghidra version.
  2. Open Ghidra and navigate to FileInstall Extensions from the Project window.
  3. In the Install Extensions dialog, click the + button (top right) to add an extension from the filesystem.
  4. Select the ReVa .zip archive and click OK.
  5. Ensure the checkbox next to ReVa is checked in the extensions list.
  6. Click OK to close the dialog.
  7. Restart Ghidra to load the extension.

The extension archive must match the Ghidra version it was built against. ReVa requires Ghidra 12.0 or later and Java 21 or later.

Plugin Activation

After installation, ReVa requires activation in two separate contexts. This is the most commonly missed step during setup.

ReVa's plugins have STABLE status, which means they auto-load in Ghidra's default tool configurations (the default CodeBrowser and the Project window) after a fresh Ghidra installation with ReVa installed. If you are using a fresh Ghidra setup, ReVa should already be active after restarting Ghidra — check the log for the server-started message before proceeding.

Users with existing or customized tool configurations may not pick up the auto-load. In that case, ReVa must be activated manually in both the Project window and the CodeBrowser as described below. Activating it in only one location will result in the server not starting or programs not being visible to MCP clients.

Step 1: Activate in the Project Window

The application-level plugin manages the MCP server lifecycle. It must be enabled in the Project window (also called the Front-End Tool):

  1. From the Ghidra Project window, select FileConfigure.
  2. In the Configure Tool dialog, locate the ReVa package (under the Common category or listed by name).
  3. Click the checkbox to enable the ReVa package.
  4. Click OK (or Done) to apply.

Once activated, the ReVa MCP server starts automatically. A log message confirms the server is running:

ReVa MCP server started on http://127.0.0.1:8080/mcp/message

Step 2: Activate in the CodeBrowser

The tool-level plugin registers open programs with the MCP server. It must be enabled in each CodeBrowser (analysis tool) instance:

  1. Open a program in the CodeBrowser.
  2. Select FileConfigure from the CodeBrowser menu.
  3. Locate the ReVa package and enable it.
  4. Click OK to apply.

Programs opened in a CodeBrowser with ReVa activated become visible to MCP clients. Programs in CodeBrowser instances without ReVa activated remain invisible to the MCP server.

To avoid repeating this step for every new CodeBrowser, save the tool configuration after activating ReVa. Use FileSave Tool to persist the setting.

MCP Client Configuration

Once ReVa is running, an MCP-compatible client must be configured to connect to the server. The connection details depend on the client.

Claude Code

Claude Code can connect to ReVa in two ways:

Option A: Direct HTTP connection (recommended for GUI mode)

Add ReVa as an MCP server in Claude Code's configuration, pointing to the HTTP endpoint:

{
  "mcpServers": {
    "ReVa": {
      "url": "http://localhost:8080/mcp/message"
    }
  }
}

Option B: Stdio bridge (recommended for CLI-only workflows)

Register the mcp-reva CLI tool, which handles server lifecycle automatically:

claude mcp add ReVa -- mcp-reva

This approach starts a dedicated ReVa server instance per Claude Code session, with its own temporary project. See CLI Installation for setup.

VS Code / Cursor

For VS Code or Cursor with MCP support, add the following to the MCP configuration:

{
  "mcpServers": {
    "ReVa": {
      "url": "http://localhost:8080/mcp/message"
    }
  }
}

Generic MCP Clients

Any MCP client that supports the streamable HTTP transport can connect to ReVa. The connection parameters are:

Parameter Default Value
Transport Streamable HTTP
Host 127.0.0.1
Port 8080
Endpoint /mcp/message
Full URL http://localhost:8080/mcp/message

If API key authentication is enabled, the client must send the key in the X-API-Key HTTP header with each request.

CLI Installation

The mcp-reva CLI tool provides stdio transport for integration with command-line AI tools. It requires Python 3.10 or later and PyGhidra.

Prerequisites

  • Python 3.10 or later
  • A Ghidra installation (12.0 or later)
  • uv package manager (recommended) or pip

Installation

Install the CLI tool using uv:

uv tool install reva-cli

Or install from the repository for development:

uv pip install -e .

Verify the installation:

mcp-reva --version

Usage

The CLI accepts the following arguments:

Argument Description
--config PATH Path to a ReVa configuration file (optional)
--verbose, -v Enable verbose logging
--version Print version and exit

When launched, the CLI initializes PyGhidra, creates a temporary Ghidra project, starts the MCP server on a random port, and bridges stdin/stdout to the HTTP server. All temporary files are cleaned up on exit.

Headless Mode Setup

Headless mode runs ReVa without the Ghidra GUI. This is useful for server deployments, batch processing, and CI/CD pipelines.

Using the Python Script

The simplest way to run headless mode is with the provided script:

python scripts/reva_headless_server.py --wait

The --wait flag keeps the server running until interrupted.

Using the Java API

For programmatic integration, use the RevaHeadlessLauncher class directly:

import pyghidra
pyghidra.start()

from reva.headless import RevaHeadlessLauncher

launcher = RevaHeadlessLauncher(
    configFile=None,
    useRandomPort=True
)
launcher.start()

if launcher.waitForServer(30000):
    print(f"Server ready on port {launcher.getPort()}")

# ... use the server ...

launcher.stop()

The launcher supports several constructors for different deployment scenarios:

  • Default — In-memory configuration, port 8080, no persistent project
  • With config file — Load settings from a .properties file
  • With random port — Avoids port conflicts in multi-instance deployments
  • With persistent project — Specify a project location and name for data that survives restarts

Configuration Options

ReVa's behavior is controlled through configuration options. In GUI mode, these are managed through Ghidra's Tool Options dialog (EditTool Options, then select ReVa Server Options). In headless mode, they are set in a .properties configuration file.

Server Options

Option Default Description
Server Port 8080 TCP port for the MCP server.
Server Host 127.0.0.1 Network interface to bind. Use 127.0.0.1 for localhost only, or 0.0.0.0 to accept connections from all interfaces.
Server Enabled true Whether the MCP server is active. Set to false to install the extension without running the server.

Security Options

Option Default Description
API Key Authentication Enabled false Require an API key for all MCP requests. Recommended when the server host is set to 0.0.0.0.
API Key (auto-generated) The API key value. If left empty when authentication is enabled, a key is generated automatically in the format ReVa-{UUID}.

Analysis Options

Option Default Description
Decompiler Timeout (seconds) 10 Maximum time allowed for each decompilation operation.
Max Decompiler Search Functions 1000 Threshold above which decompiler-based searches warn the client about performance. Does not prevent the search.

Import Options

Option Default Description
Import Timeout (seconds) 120 Maximum time for file import operations.
Import Analysis Timeout (seconds) 600 Maximum time for auto-analysis after importing a file (10 minutes).
Wait for Analysis on Import true Whether to run auto-analysis after importing a file and wait for it to complete before returning.
Import Max Depth 10 Maximum recursion depth when importing containers or archives.

Debug Options

Option Default Description
Debug Mode false Enable debug-level logging.
Request Logging Enabled false Log all MCP tool requests and responses to reva-tools.log. Useful for diagnosing client issues.

Configuration File Format

For headless and CLI modes, configuration is loaded from a standard Java .properties file. Option names are derived from the GUI option names by converting to lowercase and replacing spaces with dots. Example:

reva.server.options.server.port=8080
reva.server.options.server.host=127.0.0.1
reva.server.options.server.enabled=true
reva.server.options.api.key.authentication.enabled=false
reva.server.options.api.key=ReVa-change-this-to-a-secure-random-key
reva.server.options.debug.mode=false
reva.server.options.request.logging.enabled=false
reva.server.options.decompiler.timeout.seconds=10
reva.server.options.max.decompiler.search.functions=1000
reva.server.options.import.timeout.seconds=120
reva.server.options.import.analysis.timeout.seconds=600
reva.server.options.wait.for.analysis.on.import=true
reva.server.options.import.max.depth=10

Build from Source

To build ReVa from source, the following prerequisites are required:

  • Java 21 or later (JDK, not JRE)
  • Gradle 9.x or later
  • A Ghidra installation (12.0 or later)

Build Steps

  1. Set the GHIDRA_INSTALL_DIR environment variable to the Ghidra installation directory: 
    export GHIDRA_INSTALL_DIR=/path/to/ghidra
  2. Build the extension: 
    gradle
  3. The extension archive is created in the dist/ directory.

Use gradle directly, not ./gradlew. ReVa does not use the Gradle wrapper.

To install the extension directly into the Ghidra installation (useful during development):

gradle install

Running Tests

Run unit tests (no Ghidra environment required):

gradle test

Run integration tests (requires a graphical environment):

gradle integrationTest

Integration tests require java.awt.headless=false and run with forkEvery=1 to isolate test state. They cannot run in headless CI environments without a virtual display.

Troubleshooting

Server Does Not Start

  • Port already in use — Another process (including a previous Ghidra instance) may be using port 8080. Change the port in EditTool OptionsReVa Server Options, or stop the conflicting process.
  • Plugin not activated in Project window — The server is managed by the application-level plugin. Verify activation following the steps in Plugin Activation.
  • Server disabled in configuration — Check that Server Enabled is set to true in the Tool Options.

No Programs Visible to MCP Client

  • Plugin not activated in CodeBrowser — Each CodeBrowser window must have ReVa activated independently. See Step 2: Activate in the CodeBrowser.
  • No program open — A program must be open in the CodeBrowser for it to appear in the MCP server's program list.
  • Wrong programPath — Verify the program path using the list-open-programs tool or the ghidra://programs resource. See Understanding programPath.

Client Cannot Connect

  • Wrong URL — Ensure the client is connecting to http://localhost:8080/mcp/message (not just /mcp or the root path).
  • API key required — If API key authentication is enabled, the client must include the key in the X-API-Key header.
  • Firewall blocking connection — On some systems, the firewall may block connections to localhost. Verify the server is reachable with: 
    curl http://localhost:8080/mcp/message

Jackson Library Conflicts

If the extension fails to load with Jackson-related errors, stale library files may be cached in the extension's lib/ directory. To resolve:

  1. Remove cached libraries: 
    rm lib/*.jar
  2. Rebuild the extension: 
    gradle

CLI Mode Issues

  • PyGhidra initialization fails — Ensure GHIDRA_INSTALL_DIR is set and points to a valid Ghidra installation.
  • Stdio bridge hangs — The CLI requires a clean stdin/stdout channel. Ensure no other process is writing to stdout (e.g., shell initialization scripts printing messages).
  • Slow startup — Initial startup takes 4–7 seconds while PyGhidra and the Ghidra framework initialize. This is normal.
