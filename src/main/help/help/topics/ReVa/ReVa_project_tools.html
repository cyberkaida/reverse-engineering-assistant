ReVa Project Management Tools

ReVa provides a set of project management tools that allow AI assistants to discover, inspect, and manage programs within a Ghidra project. These tools are the starting point for most analysis workflows — you need to identify a program before you can analyze it.

All project tools use the programPath field for program identification. See Program Identification in the Architecture & Concepts page for details on how programPath works.

get-current-program

Returns metadata about the program currently active in the focused CodeBrowser window. This is typically the first tool an AI assistant calls to identify which program to work with.

This tool is only available in GUI mode (Assistant mode). It is not registered in headless or CLI/stdio mode, since there is no active CodeBrowser window.

Parameters

This tool takes no parameters.

Response Fields

Field Type Description
programPath string The Ghidra project pathname (e.g., /Hatchery.exe). Use this value as the programPath parameter in all other tools.
language string The processor language ID (e.g., x86:LE:64:default)
compilerSpec string The compiler specification ID (e.g., gcc, windows)
creationDate long Timestamp when the program was imported into Ghidra
modificationDate long Timestamp of the last modification
sizeBytes long Total size of the program's memory blocks in bytes
symbolCount int Number of symbols in the symbol table
functionCount int Number of defined functions
isReadOnly boolean Whether the program is opened in read-only mode

Example Response

{
  "programPath": "/Hatchery.exe",
  "language": "x86:LE:64:default",
  "compilerSpec": "windows",
  "creationDate": 1709251200000,
  "modificationDate": 1709337600000,
  "sizeBytes": 245760,
  "symbolCount": 1523,
  "functionCount": 387,
  "isReadOnly": false
}

list-project-files

Lists files and folders within the Ghidra project. This tool lets you browse the project structure to discover available programs, even those that are not currently open in any CodeBrowser window. Programs listed here can be opened on demand by passing their programPath to any analysis tool.

Parameters

Parameter Type Required Default Description
folderPath string Yes — Path to the folder to list. Use "/" for the project root.
recursive boolean No false When true, lists all files and folders recursively within the specified path.

Response Fields

The response contains metadata about the folder and an array of items:

Metadata:

Field Type Description
folderPath string The requested folder path
folderName string The name of the folder
isRecursive boolean Whether the listing is recursive
itemCount int Total number of items returned

File items:

Field Type Description
programPath string The program's project pathname (use this with other tools)
type string "file"
contentType string The Ghidra content type (e.g., "Program")
lastModified long Last modification timestamp
readOnly boolean Whether the file is read-only
versioned boolean Whether the file is under version control
checkedOut boolean Whether the file is currently checked out
programLanguage string Processor language (optional, for program files)
executableMD5 string MD5 hash of the original executable (optional)

Folder items:

Field Type Description
folderPath string The folder's project pathname
type string "folder"
childCount int Number of items contained in this folder

Example: List Root Folder

Request: { "folderPath": "/" }

{
  "folderPath": "/",
  "folderName": "",
  "isRecursive": false,
  "itemCount": 3,
  "items": [
    {
      "programPath": "/Hatchery.exe",
      "type": "file",
      "contentType": "Program",
      "lastModified": 1709337600000,
      "readOnly": false,
      "versioned": true,
      "checkedOut": true,
      "programLanguage": "x86:LE:64:default",
      "executableMD5": "a1b2c3d4e5f6..."
    },
    {
      "folderPath": "/malware",
      "type": "folder",
      "childCount": 5
    },
    {
      "programPath": "/bootloader.elf",
      "type": "file",
      "contentType": "Program",
      "lastModified": 1709251200000,
      "readOnly": false,
      "versioned": false,
      "checkedOut": false,
      "programLanguage": "ARM:LE:32:v7"
    }
  ]
}

Use "recursive": true to get a complete listing of all programs in the project at once. For large projects with many folders, start with a non-recursive listing of "/" and then explore specific folders.

list-open-programs

Lists all programs currently open across all CodeBrowser windows. This is useful when multiple programs are open simultaneously and you need to see which ones are available for analysis.

This tool is only available in GUI mode. In headless or CLI/stdio mode, use list-project-files instead.

Parameters

This tool takes no parameters.

Response Fields

The response contains a count and an array of program objects, each with the same fields as get-current-program:

Field Type Description
count int Number of open programs
programs array Array of program metadata objects (same fields as get-current-program)

Example Response

{
  "count": 2,
  "programs": [
    {
      "programPath": "/Hatchery.exe",
      "language": "x86:LE:64:default",
      "compilerSpec": "windows",
      "sizeBytes": 245760,
      "symbolCount": 1523,
      "functionCount": 387,
      "isReadOnly": false
    },
    {
      "programPath": "/malware/sample_001.bin",
      "language": "x86:LE:32:default",
      "compilerSpec": "gcc",
      "sizeBytes": 81920,
      "symbolCount": 256,
      "functionCount": 42,
      "isReadOnly": false
    }
  ]
}

checkin-program

Commits the current state of a program to Ghidra's version control system with a descriptive message. This is analogous to a git commit — it creates a versioned snapshot of the program that can be restored later. If the program is not yet under version control, it will be added automatically.

Parameters

Parameter Type Required Default Description
programPath string Yes — The program to check in
message string Yes — A commit message describing the changes (e.g., "Renamed networking functions")
keepCheckedOut boolean No false When true, the program remains checked out after the commit, allowing continued editing. When false, the program is checked in and must be checked out again for further modifications.

Response Fields

Field Type Description
success boolean Whether the operation succeeded
action string The action taken: "added_to_version_control" (new file), "checked_in" (existing versioned file), or "saved" (non-shared project)
programPath string The program that was checked in
message string The commit message
keepCheckedOut boolean Whether the program remains checked out
isVersioned boolean Whether the program is now under version control
isCheckedOut boolean Whether the program is currently checked out
info string Additional information (optional)

Example: Check In with Continued Editing

Request:

{
  "programPath": "/Hatchery.exe",
  "message": "Renamed networking functions and fixed socket struct types",
  "keepCheckedOut": true
}

Response:

{
  "success": true,
  "action": "checked_in",
  "programPath": "/Hatchery.exe",
  "message": "Renamed networking functions and fixed socket struct types",
  "keepCheckedOut": true,
  "isVersioned": true,
  "isCheckedOut": true
}

Use "keepCheckedOut": true when you want to create a checkpoint during a long analysis session. This saves your progress while allowing you to continue making changes without interruption.

list-programs MCP Resource

In addition to the tools above, ReVa exposes an MCP resource for program discovery. Resources are read-only data endpoints that MCP clients can subscribe to for updates.

Property Value
Resource URI ghidra://programs
Resource Name open-programs
MIME Type text/plain

The resource returns a JSON array of metadata for all currently open programs:

[
  {
    "programPath": "/Hatchery.exe",
    "language": "x86:LE:64:default",
    "compilerSpec": "windows",
    "sizeBytes": 245760
  }
]

MCP resources are a different mechanism from MCP tools. Resources provide data that clients can read and subscribe to, while tools are actions that clients invoke. Not all MCP clients support resources — if your client only supports tools, use list-open-programs or list-project-files instead.

Common Workflows

Discovering Programs

The typical workflow for discovering which programs are available:

  1. Call get-current-program to see the active program (GUI mode)
  2. If you need a different program, call list-open-programs to see all open programs
  3. If the program is not open, call list-project-files with "folderPath": "/" to browse the project
  4. Use the programPath from any of these responses with analysis tools

Checkpointing Your Work

To save progress during a long analysis session:

  1. Perform analysis (rename functions, fix types, add comments, etc.)
  2. Call checkin-program with a descriptive message and "keepCheckedOut": true
  3. Continue working — the checkpoint is saved and you can restore to it if needed
  4. When finished, call checkin-program with "keepCheckedOut": false to finalize

Working in Headless/CLI Mode

In headless and CLI modes, the GUI-only tools are not available. Instead:

  1. Use list-project-files to discover available programs
  2. Use import-file to import new binaries into the project
  3. Pass the programPath to analysis tools — programs are opened on demand from the project files
