ReVa - Virtual Table & Constant Tools

These tools support advanced binary analysis tasks: analyzing C++ virtual function tables (vtables) and searching for constant values used in instructions. Vtable tools are essential for understanding class hierarchies and virtual method dispatch in C++ binaries. Constant tools help identify magic numbers, error codes, buffer sizes, and other significant values.

Virtual Table Tools

Tools for analyzing virtual function tables (vtables), finding callers of virtual methods, and discovering which classes implement a given virtual function.

analyze-vtable

Analyzes a virtual function table at a given address, returning the list of function pointers with their slot indices and byte offsets. Supports both structure-based analysis (when a structure data type is defined at the address) and raw pointer probing as a fallback.

Analysis Strategies

  1. Structure-based (preferred) — If a Structure data type is defined at the vtable address, the tool uses its components for accurate field names and types. This produces the most reliable results.
  2. Memory-based (fallback) — If no structure is defined, the tool reads raw pointers from memory and probes each one to determine if it points to a function.

Vtable Boundary Detection

In memory-based mode, the tool uses heuristics to detect the end of the vtable:

  • Requires a minimum of 2 consecutive function pointers to identify a valid vtable.
  • Allows up to 1 non-function pointer (which may be an RTTI pointer, common in MSVC).
  • Stops reading after 2 consecutive non-function pointers, indicating the end of the vtable.
  • Initial probe examines the first 5 entries to validate the vtable.

find-vtable-callers

Finds all indirect call instructions that could be dispatching a virtual method through its vtable slot. Given a function that appears in a vtable, this tool identifies all indirect calls with a matching byte offset. Offset extraction patterns are optimized for x86/x64 architectures.

If vtableAddress is not provided, the tool automatically searches for vtables containing the function. If no vtables are found, an error is returned indicating that the function may not be virtual or vtables have not been identified yet.

Constant Search Tools

Tools for finding constant values (immediate operands) used in instructions. These tools scan the program's instruction listing to locate magic numbers, error codes, buffer sizes, and other significant constant values.

All constant search tools share common limits: a 120-second timeout and a 2,000,000 instruction scan limit. For very large binaries, the instructionLimitReached flag in the response indicates whether the scan was truncated.

Value Matching

The tool checks both signed and unsigned interpretations of each scalar operand. This means searching for -1 will find instructions using 0xFFFFFFFF (32-bit) or 0xFFFFFFFFFFFFFFFF (64-bit), and vice versa. Each instruction is counted only once, even if multiple operands contain the target value.

Range Checking

The tool checks the signed interpretation first (the common use case for ranges), then checks the unsigned interpretation using proper unsigned comparison for large values. A match in either interpretation is accepted.

list-common-constants

Finds the most frequently used constant values in the program and identifies known magic numbers. By default, small values (0–255) and -1 are filtered out as noise, since they are overwhelmingly common in most binaries.

Known Constant Recognition

The tool recognizes the following categories of constants (checked in priority order):

CategoryExamplesDescription
Common sizes 0x400, 0x1000, 0x2000, 0x10000, 0x100000, 0x400000 1 KB, 4 KB (page size), 8 KB, 64 KB, 1 MB, 4 MB
Integer limits 0x7fffffff, 0x80000000, 0x7fffffffffffffff INT32_MAX, INT32_MIN/sign bit, INT64_MAX
Bit masks 0xffff, 0xffffff, 0xffffffff 16-bit mask, 24-bit mask, 32-bit mask / -1
Powers of 2 Any value where (value & (value - 1)) == 0 Reported as "2^n (power of 2)"
HTTP status codes 100–599 "Possible HTTP status code"
Windows errors 0x80000000–0x8000ffff "Possible HRESULT/NTSTATUS"
Magic numbers 0x5a4d, 0x4550, 0x464c457f, 0xdeadbeef, 0xcafebabe, 0xfeedface MZ header, PE signature, ELF magic, debug marker, Java class/Mach-O FAT, Mach-O 32/64-bit

The noise filter (excluding 0–255 and -1) is applied to both signed and unsigned interpretations. If either interpretation exceeds the threshold, the value passes the filter. The uniqueFunctions field is particularly useful for distinguishing constants that are used broadly across the program from those concentrated in a single function.

Tips & Best Practices

  • Use analyze-vtable to inspect a known vtable, then find-vtable-callers to find where those virtual methods are called.
  • Use find-vtables-containing-function to discover class hierarchies — if a function appears in multiple vtables, those classes likely share a common base.
  • For the best vtable analysis results, define a structure at the vtable address first using the Structure Tools.
  • Start with list-common-constants for a broad overview of significant constants, then use find-constant-uses to trace specific values of interest.
  • Use find-constants-in-range with HTTP status code ranges (400–599) to identify error handling code, or with HRESULT ranges to find Windows error paths.
  • When searching for a constant, remember that both signed and unsigned interpretations are checked — searching for -1 will also find 0xFFFFFFFF.
  • The uniqueFunctions count in list-common-constants helps distinguish pervasive constants (alignment, buffer sizes) from localized ones (algorithm keys, protocol markers).
