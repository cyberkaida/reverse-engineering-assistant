ReVa Cross-References & Data Flow Tools

The ReVa cross-reference and data flow tools provide MCP-accessible interfaces for finding references between code and data locations, tracing how values propagate through a program, and identifying where variables are read or written. These tools are essential for understanding data dependencies and code relationships.

find-cross-references

Find incoming and outgoing cross-references for an address or symbol. This unified tool combines reference direction, type filtering, pagination, and optional decompilation context into a single query. Use it to understand how a function, variable, or address is used throughout the program.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
location string Yes — Address or symbol name to find references for (e.g., "0x00400123", "main", "FUN_00401000").
direction string No "both" Reference direction to query: "to" (incoming references pointing to this location), "from" (outgoing references from this location), or "both" (both directions).
includeFlow boolean No true Include flow references: calls, jumps, and branches.
includeData boolean No true Include data references: reads and writes.
includeContext boolean No false Include decompilation context snippets around each code reference. Only applies to flow references.
contextLines integer No 2 Number of decompilation lines to show before and after each reference when includeContext is true.
offset integer No 0 Starting offset for pagination (0-based).
limit integer No 100 Maximum number of references to return per direction. Hard cap: 1,000.

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • location — Resolved location details:
    • address — Resolved hex address.
    • symbol — Symbol name (if any).
    • symbolType — Type of symbol (e.g., "FUNCTION").
    • function — Containing function name.
    • isFunctionEntry — Whether the address is a function entry point.
  • referencesTo — (If direction is "to" or "both") Array of incoming reference objects.
  • referencesFrom — (If direction is "from" or "both") Array of outgoing reference objects.

Each reference object contains:

  • fromAddress, toAddress — Source and destination addresses.
  • referenceType — Ghidra reference type string (e.g., "CALL", "DATA", "JUMP").
  • isPrimary — Whether this is the primary reference at its address.
  • operandIndex — Instruction operand index.
  • sourceType — How the reference was created (e.g., "USER_DEFINED", "ANALYSIS").
  • Classification flags: isCall, isJump, isData, isRead, isWrite.
  • fromSymbol, toSymbol — Symbol information at each end (name and type).
  • fromFunction or toFunction — Function context including name, entry address, and optionally line number and context snippet (when includeContext is true).
  • pagination — Object with offset, limit, totalToCount, totalFromCount, hasMoreTo, and hasMoreFrom.

Behavior Notes

  • For function entry addresses, outgoing references are collected across the entire function body, not just the entry point instruction.
  • Stack and register addresses cannot have incoming references; these are automatically filtered.
  • Context snippets are only generated for flow references (calls, jumps), not for data references.
  • Individual reference processing failures are logged but do not abort the operation.

To find all callers of a function, use direction="to" with includeFlow=true and includeData=false. To find all data readers/writers, use includeFlow=false and includeData=true.

trace-data-flow-backward

Trace where a value at a specific address comes from by following data dependency chains backward through the program's SSA (Static Single Assignment) representation. This tool reveals the origins of a value — whether it comes from a constant, function parameter, global variable, or memory load.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
address string Yes — Address within a function to trace backward from. Must be an address with an instruction that defines or uses a value.

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • function, functionAddress — The containing function.
  • startAddress — The address tracing started from.
  • direction — Always "backward".
  • operationCount — Number of P-code operations in the slice.
  • operations — Array of P-code operation objects, each with:
    • address — Address of the operation.
    • opcode — P-code opcode name (e.g., "COPY", "LOAD", "INT_ADD").
    • output — Output varnode with type (register, constant, temporary, memory, other), offset, size, and optional variableName.
    • inputs — Array of input varnodes with the same fields.
  • terminators — Array of data origin points. Each terminator has a type and type-specific fields:
    • CONSTANT — A literal constant value. Fields: value.
    • PARAMETER — A function parameter. Fields: name, slot.
    • GLOBAL — A global variable. Fields: name, address.
    • MEMORY_LOAD — A load from memory. Fields: address.

Limits and Configuration

  • Maximum slice size: 500 P-code operations. Tracing stops at this limit to prevent memory exhaustion.
  • Decompiler timeout: 30 seconds by default (configurable via ConfigManager).
  • The tool uses Ghidra's DecompilerUtils.getBackwardSliceToPCodeOps() for SSA-based backward slicing.
  • The decompiler is configured with syntax tree output enabled and C code output disabled for optimal performance.

The address must point to an instruction that defines or uses a value. If the address has no data flow information, the error message will suggest trying an address with an instruction that uses or defines a value.

trace-data-flow-forward

Trace where a value at a specific address flows to by following data dependency chains forward through the SSA representation. This tool reveals the sinks of a value — whether it is stored to memory, passed to a function call, or returned from the function.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
address string Yes — Address within a function to trace forward from. Must be an address with an instruction that defines or uses a value.

Return Value

Returns a JSON object with the same structure as trace-data-flow-backward, except:

  • direction — Always "forward".
  • terminators — Array of data sink points instead of origins. Terminator types are:
    • MEMORY_STORE — A store to memory. Fields: address.
    • FUNCTION_CALL — A call to another function. Fields: address, targetAddress.
    • RETURN — A function return statement. Fields: address.

All other fields (operations, operationCount, etc.) follow the same format.

Limits and Configuration

  • Maximum slice size: 500 P-code operations (same as backward tracing).
  • Decompiler timeout: 30 seconds by default.
  • Uses Ghidra's DecompilerUtils.getForwardSliceToPCodeOps() for SSA-based forward slicing.

Combine trace-data-flow-backward and trace-data-flow-forward on the same address to see the complete data lifecycle: where a value comes from and where it goes.

find-variable-accesses

Find all locations where a specific variable is read or written within a function. This tool identifies every access point for a named variable, distinguishing between reads and writes, and reporting the P-code operation at each access.

Parameters

Parameter Type Required Default Description
programPath string Yes — Path in the Ghidra Project to the program.
functionAddress string Yes — Address of the function, or an address within it. The containing function is resolved automatically.
variableName string Yes — Name of the variable to track. Must match a variable visible in the decompilation output.

Return Value

Returns a JSON object containing:

  • programPath — Program path.
  • function, functionAddress — The containing function.
  • variableName — The variable that was searched.
  • variableType — Classification of the variable: "parameter", "local", "global", or "unknown".
  • dataType — The variable's data type.
  • accessCount — Total number of access points found.
  • accesses — Array of access objects, each with:
    • address — Address of the access.
    • accessType — Either "WRITE" (variable definition) or "READ" (variable use).
    • operation — The P-code operation name at that access (e.g., "COPY", "INT_ADD", "CALL", "STORE").

Variable Resolution

The tool searches for the variable among all high-level symbols in the function, including local variables (HighLocal), parameters (HighParam), and global variables (HighGlobal). If the variable is not found, the error message lists all available variable names in the function.

Access detection works as follows:

  • WRITE access is recorded at any P-code operation where the variable is the defined (output) varnode.
  • READ access is recorded at any P-code operation where the variable is used as an input (descendant) varnode.
  • Duplicate addresses are automatically deduplicated.

Use this tool after get-decompilation to understand how a specific variable is used throughout a function. The access addresses can then be passed to trace-data-flow-backward or trace-data-flow-forward for deeper analysis.
